StrOutputParser

Step 3 in the classic pipe — turns whatever the model returned into a plain str you can print, log, or pass on.

Job (help-desk story)

Formatting desk at the end of the line. The specialist (LLM) hands back a raw reply; this step strips wrappers and always delivers one clean string to the customer.

Problem without it

  • ChatWatsonx and chat-style chains often return AIMessage objects — print(chain.invoke(...)) shows object repr, not answer text.
  • Downstream code expects str; without a parser you add manual .content access in every script.
  • Swapping model types (text LLM → chat LLM) breaks callers unless the last step normalizes output.
  • Structured outputs (JSON, lists) need different parsers — this one is only for plain text.

Object lifecycle

  1. Build — parser = StrOutputParser() — no config, stateless.
  2. Configure — None. Wire as last step: prompt | llm | parser.
  3. Invoke — You never call parser.invoke alone in labs; the chain passes LLM output into it automatically.
  4. Output — Always str. That becomes the return value of chain.invoke(dict).

Data flow

chain.invoke({"content": passage, "question": q})
        │
        ▼
   PromptTemplate  →  filled prompt
        │
        ▼
   WatsonxLLM      →  str (text API) or AIMessage (chat API)
        │
        ▼
 StrOutputParser   →  str  ← chain return type
        │
        ▼
   print(result)   # human-readable sentence

Minimal example

from langchain_core.output_parsers import StrOutputParser

chain = prompt | llm | StrOutputParser()
text = chain.invoke({"content": passage})  # str

Full lab (your playground)

from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import StrOutputParser
from watson_llm import make_watsonx_llm

llm = make_watsonx_llm()
template = "Answer the {question} based on the {content}."
prompt = PromptTemplate.from_template(template)

qa_chain = prompt | llm | StrOutputParser()
print(qa_chain.invoke({
    "question": "Which planets are rocky?",
    "content": "Mercury, Venus, Earth, Mars are rocky...",
}))

Traps (from your labs)

  • Dropping parser on chat pipes — Lab 20 uses joke_prompt | llama_llm without parser. Works while output looks like text; add StrOutputParser() before production or you get AIMessage surprises.
  • Using StrOutputParser for JSON — Lab 21 needs JsonOutputParser or CommaSeparatedListOutputParser. StrOutputParser leaves JSON as a string you must json.loads yourself.
  • Parser in the wrong position — StrOutputParser() | llm is invalid. Parser is always after the LLM.

Recall drills

Where does StrOutputParser sit in prompt | llm | ????
Last step — after the LLM. It receives model output and returns str.
When is it optional with WatsonxLLM but required with ChatWatsonx?
Text LLM often already returns something printable; chat LLM returns message objects — parser extracts plain text consistently.
What parser do you use for Lab 21 Part A (Joke setup + punchline)?
JsonOutputParser with a Pydantic class — not StrOutputParser. See Module 11.
