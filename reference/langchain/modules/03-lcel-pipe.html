LCEL pipe

Wire Runnables with | — fixed steps, one invoke runs them all. This is the spine every other LangChain pattern hangs on.

Job (help-desk story)

Ticket intake → specialist → formatted reply. The customer fills one form (invoke dict); the pipeline runs every desk in order without you hand-carrying strings between steps.

Problem without it

  • You call prompt.format(), then llm.invoke(), then parse — easy to skip a step or pass the wrong type.
  • Each step has a different calling convention; pipes unify everything behind Runnable.invoke.
  • When you add RAG or parsers later, you swap step 2 or 3 — the outer chain.invoke call stays the same.

Object lifecycle

  1. Build — At setup: chain = prompt | llm | parser. Type is RunnableSequence. Steps are frozen; model does not choose them at runtime.
  2. Configure — Each step is configured separately (PromptTemplate string, WatsonxLLM params). The pipe only wires references.
  3. Invoke — chain.invoke({"key": value}). Dict keys must match every {hole} in the prompt template.
  4. Output — Return type = last step’s output. With StrOutputParser → str. With JsonOutputParser → dict.

Data flow

invoke({"adjective": "funny", "content": "chickens"})
        │
        ▼
┌───────────────────┐
│  PromptTemplate   │  dict → PromptValue (filled template)

Minimal example

chain = prompt | llm | StrOutputParser()
chain.invoke({"content": "long passage here"})

Full lab (your playground)

from langchain_core.prompts import PromptTemplate
from langchain_core.output_parsers import StrOutputParser
from watson_llm import make_watsonx_llm

llm = make_watsonx_llm()
template = "Tell me a {adjective} joke about {content}."
prompt = PromptTemplate.from_template(template)
chain = prompt | llm | StrOutputParser()

print(type(chain))  # RunnableSequence
print(chain.invoke({"adjective": "funny", "content": "chickens"}))

Traps (from your labs)

  • Wrong LLM in pipe — watson_helper.model (ModelInference) cannot sit after |. Use make_watsonx_llm() from 07.
  • format() vs invoke() — prompt.format(adjective="x") is a manual test only. In a pipe you always call chain.invoke({"adjective": "x", ...}).
  • Chain ≠ agent — If the model needs to choose tools or loop, you are past LCEL pipes (see Lab 35). Pipes do not think; they execute.

Recall drills

What type is prompt | llm | parser, and when are steps chosen?
RunnableSequence. Steps are fixed when you build the chain — not chosen by the model at invoke time.
What do you pass to chain.invoke(...) for a PromptTemplate first step?
A dict whose keys match every {variable} in the template string. Missing key → error before the LLM runs.
Why keep StrOutputParser even when WatsonxLLM already returns a string?
So the last step always returns str when you swap to ChatWatsonx (message objects) or add a different model — pipe contract stays stable.
