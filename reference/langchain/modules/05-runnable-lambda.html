RunnableLambda

Wrap any Python function as a pipe step — Coursera’s way to call prompt.format(**variables) inside LCEL.

Job (help-desk story)

Custom prep desk before the specialist. Something the template object cannot do alone — reshape the invoke dict, strip text, merge fields — runs as step 1 instead of PromptTemplate.

Problem without it

  • Coursera notebooks teach def format_prompt(variables): return prompt.format(**variables) — you need a Runnable wrapper to put that in a pipe.
  • Some transforms are awkward in template strings (heavy string ops, conditional logic).
  • Without lambda, you might format outside the chain and break the single invoke(dict) pattern.

But: for plain hole-filling, PromptTemplate is already a Runnable — lambda is redundant. See direct equivalent below.

Legitimate technical uses (not just Coursera)

Coursera’s format(**variables) wrapper is teaching habit from pre-LCEL notebooks. In 1.x, prompt | llm replaces it. RunnableLambda still matters when step 1 (or any step) must be real Python:

Use caseWhy template alone failsSketch
Normalize / clean input Strip whitespace, truncate to token budget, regex scrub lambda d: prompt.format(content=d["content"].strip()[:4000])
Reshape invoke dict API gives {"q": ...} but template expects {question} lambda d: prompt.format(question=d["q"])
Merge multiple fields Build one string from several keys + logic lambda d: f"Q: {d['q']}\nContext: {d['ctx'][:2000]}"
Conditional prep Different prefix if d["lang"] == "fr" if/else inside function — templates are weak here
Post-LLM cleanup Strip markdown fences before JsonOutputParser llm | RunnableLambda(strip_fences) | json_parser
Bridge legacy code Existing def build_prompt(row): ... not rewritten yet Wrap once, keep pipe uniform

Rule of thumb: if it fits in one PromptTemplate string with {holes} → no lambda. If you need Python control flow or string ops → lambda (or a small named function) is the right tool.

Practical example (4-stage pipe)

Chains are not limited to 3 steps. Lambda can sit after prompt — it receives PromptValue, not the invoke dict.

qa_prompt = PromptTemplate.from_template(
    "Context:\n{context}\n\nQuestion: {question}\n\nAnswer in one short sentence:"
)

def cap_filled_prompt(prompt_value) -> str:
    text = prompt_value.to_string()
    marker = "\n\nQuestion:"
    idx = text.find(marker)
    if idx == -1 or len(text) <= 500:
        return text[:500] if len(text) > 500 else text
    tail = text[idx:]  # keep Question + Answer instruction
    head = text[:idx][: 500 - len(tail) - 25]
    return head + "\n… [context truncated]" + tail

# 4 stages — not just prompt | llm | parser
chain = (
    qa_prompt
    | RunnableLambda(cap_filled_prompt)
    | llm
    | StrOutputParser()
)

chain.invoke({"question": q, "context": huge_document})

Run: playground/langchain/14b.runnable_lambda.practical.py (Part A prints lengths before/after cap).

Practical example (post-LLM cleanup)

Lambda after the LLM — strip markdown fences so the next step gets plain text.

def strip_markdown_fences(text: str) -> str:
    t = text.strip()
    if not t.startswith("```"):
        return t
    lines = t.splitlines()[1:]
    if lines and lines[-1].strip() == "```":
        lines = lines[:-1]
    return "\n".join(lines).strip()

chain = prompt | llm | RunnableLambda(strip_markdown_fences) | StrOutputParser()  # 4 stages
# or 5: prompt | RunnableLambda(cap) | llm | RunnableLambda(strip) | StrOutputParser()

Part B in 14b.runnable_lambda.practical.py — local demo, no watsonx call.

Object lifecycle

  1. Build — Define a function fn(input) → output, then RunnableLambda(fn). fn must accept what the previous step would pass (usually the invoke dict).
  2. Configure — Function closes over objects (e.g. product_review_prompt defined above).
  3. Invoke — chain.invoke({"review": text}) → lambda receives {"review": text} as variables.
  4. Output — Whatever your function returns. For Coursera format pattern → str → LLM → StrOutputParser → str.

Data flow (Coursera review pattern)

Traps (from your labs)

  • RunnableLambda(prompt) — Pass a function, not the PromptTemplate instance. Coursera always wraps format_review_prompt.
  • Forgetting **variables — format(variables) fails; need prompt.format(**variables) so dict keys become keyword args matching {review}.
  • Lambda when prompt | llm suffices — Labs 08–13 work without lambda. Extra step = extra confusion unless you need real Python prep.
  • Output format drift — Lab 14: model may add extra sections (e.g. Recommendation). Lower TEMPERATURE (0.2) or tighten template bullets.

Recall drills

What does format_review_prompt(variables) receive when you call chain.invoke({"review": text})?
The dict {"review": text} as the single argument variables.
Rewrite Lab 14 without RunnableLambda — one line for the chain.

When is RunnableLambda worth keeping (not replacing with prompt | llm)?
When step 1 needs real Python — strip/normalize text, rename keys, merge multiple inputs — not just template hole fill.
