Document + loaders

RAG starts here: turn files and URLs into Document objects the rest of the pipeline can chunk, embed, and retrieve.

Key take

StepWhatLab
EnvelopeDocument(page_content=..., metadata={...})23
PDF loadPyPDFLoader(url).load() → list, one doc per page24
Web loadWebBaseLoader(url).load() → usually one big doc25

Memory hook: content + metadata · loader.load() → list[Document]. No LLM yet — ingest only.

Job (help-desk story)

Mailroom. Loaders fetch raw material (PDF pages, web HTML) and slip each piece into a standard envelope (Document) with a label (metadata) before indexing or RAG.

Problem without it

  • You paste whole PDFs into prompts — context overflow and hallucination.
  • No source / page metadata — cannot cite where an answer came from.
  • Every script reinvents file I/O instead of a shared Document shape for splitters and vector stores.

Lifecycle

  1. Hand-build (Lab 23) — learn the envelope: page_content + metadata.
  2. Pick loader — PDF → PyPDFLoader; web → WebBaseLoader.
  3. Load — documents = loader.load() → list[Document].
  4. Next — pass list to splitter (13), then embeddings (14).

Minimal example (Lab 23)

from langchain_core.documents import Document

doc = Document(
    page_content="Python emphasizes readability.",
    metadata={"source": "About Python"},
)
print(doc.page_content, doc.metadata)

Traps

  • Expecting one Document for whole PDF — PyPDFLoader returns one per page.
  • No network — Labs 24–25 need URL fetch; Lab 23 is offline-safe.
  • Skipping Lab 23 — hand-built Document teaches the shape loaders produce.
  • langchain_community deprecation warnings — OK for learning; imports still work in your venv.

Recall drills

Two fields on every Document?
page_content (text) and metadata (dict — source, page, custom keys).
PDF loader vs web loader — output list length?
PDF: usually many (one per page). Web: often one document for the whole page.
What method turns a loader into documents?
loader.load() → list[Document].
