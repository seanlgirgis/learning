Chat message types

Same make_watsonx_llm() from 07 — but instead of one plain string, pass a list of role-tagged messages to invoke.

Two invoke shapes (key take)

Still Route B (watson_llm). Only the input changes:

ShapeInvokeLabWhen
String llm.invoke("Hello") 16 Quick one-off question — no roles
Message list llm.invoke([SystemMessage(...), HumanMessage(...)]) 17 System rules, user text, fake prior turns

Memory hook: string → Lab 16 · list of messages → Lab 17. Output is still str from WatsonxLLM.

Job (help-desk story)

Email thread with labeled headers: From: System (rules), From: User (question), optional From: Assistant (a prior reply you paste in). You hand the whole thread to the model each time — not a single blob of text.

Problem without message types

  • One string mixes “you are a book bot” and “what should I read?” — model may ignore instructions or role-play badly.
  • Fake multi-turn chat (user → AI → user) is awkward as one string with manual labels.
  • Later, ChatPromptTemplate (Module 10) and chat models expect message objects — Lab 17 teaches the raw shape first.

Lifecycle (build the list, invoke once)

  1. LLM — llama_llm = make_watsonx_llm() (same as Lab 16).
  2. Import — SystemMessage, HumanMessage, AIMessage from langchain_core.messages.
  3. Build list — Order matters: usually System first, then Human / AI / Human…
  4. Invoke — llama_llm.invoke([...]) → str. Not memory — you pass the full list every call.

Flow

make_watsonx_llm()   ← Route B (07)
        │
        ▼
┌─────────────────────────────────────────┐
│  [ SystemMessage("You are a book bot")  │
│    HumanMessage("Mystery novels?")     ]│  ← list, not one string
└─────────────────┬───────────────────────┘
                  ▼
           llm.invoke([...])
                  ▼
            "Try Agatha Christie."   ← str

Optional fake history: insert AIMessage between two HumanMessage lines so the model sees a prior assistant reply you supplied.

Three message types

TypeRoleIf wrong then…
SystemMessage Instructions / persona — usually first Model drifts, rambles, or ignores tone you wanted
HumanMessage User question or follow-up Model may answer as if it wrote the “user” line (role confusion)
AIMessage Prior assistant line you supply (fake history) Not model output from last call — you build it manually for context 
from langchain_core.messages import HumanMessage, SystemMessage, AIMessage

Minimal example

from watson_llm import make_watsonx_llm
from langchain_core.messages import SystemMessage, HumanMessage

llm = make_watsonx_llm()
print(llm.invoke([
    SystemMessage(content="One short sentence only."),
    HumanMessage(content="Recommend a mystery book."),
]))

Full lab (Lab 17 — three patterns)

from watson_llm import GenParams, make_watsonx_llm
from langchain_core.messages import HumanMessage, SystemMessage, AIMessage

llama_llm = make_watsonx_llm({GenParams.MAX_NEW_TOKENS: 80})

# 1 — system + human
llama_llm.invoke([
    SystemMessage(content="Book recommender. One short sentence."),
    HumanMessage(content="I enjoy mystery novels, what should I read?"),
])

# 2 — fake multi-turn (AIMessage = prior reply you paste in)
llama_llm.invoke([
    SystemMessage(content="Fitness bot. One short sentence."),
    HumanMessage(content="I like HIIT, what should I do?"),
    AIMessage(content="Try a CrossFit class."),
    HumanMessage(content="How often should I attend?"),
])

# 3 — human only (no system)
llama_llm.invoke([HumanMessage(content="What month follows June?")])

Traps (from Lab 17)

  • Not memory — each invoke is fresh. Prior turns only exist if you put them in the list (e.g. AIMessage).
  • String vs list — invoke("hello") works (Lab 16); invoke([HumanMessage(...)]) is the message-list shape (Lab 17). Don’t mix up which lab teaches which.
  • Rambling / role-play — WatsonxLLM text API may ignore “one sentence.” Tight SystemMessage + lower MAX_NEW_TOKENS (Lab 17 uses 80).
  • Wrong route — messages go to make_watsonx_llm(), not watson_helper.model.generate().

Recall drills

Lab 16 vs Lab 17 — what do you pass to invoke?
Lab 16: a string. Lab 17: a list of SystemMessage / HumanMessage / AIMessage.
What is AIMessage for in Lab 17 example 2?
A fake prior assistant reply you insert so the model sees user → assistant → follow-up user — not automatic memory.
Three message types and one-line job of each?
SystemMessage = rules/persona · HumanMessage = user text · AIMessage = prior assistant line you supply.
