Retrievers + RAG

Retrieve relevant chunks, augment the prompt, generate an answer — the payoff of modules 12–15.

Key take — three levels

LevelAPILab
Searchretriever = vector_store.as_retriever(search_kwargs={"k": 3})30
RAG chainRetrievalQA.from_chain_type(llm, retriever=..., chain_type="stuff")31
Parent retrieverSmall chunks search → large parent Document returned32

Memory hook: retriever finds · RetrievalQA finds + answers · langchain_classic.chains for QA.

Job (help-desk story)

Research librarian + writer. Librarian (retriever) pulls three relevant passages; writer (LLM) answers only from that packet — not from memory of the whole internet.

Flow (RetrievalQA — Lab 31)

Question: "what is this paper discussing?"
        │
        ▼
   retriever.invoke(question)  →  top-k Document chunks
        │
        ▼
   stuff chunks into prompt + LLM
        │
        ▼
   answer str (grounded in PDF)

Traps

  • Wrong PDF source — Lab 31 uses LangChain paper PDF; random URL → weak answers.
  • langchain_classic — RetrievalQA import path in 1.x venv, not bare langchain.chains.
  • Retriever without ingest — must build vector store first (29 → 30 → 31).
  • Parent vs flat retriever — Lab 32 QA section may use flat Chroma; parent retriever is a separate pattern in same file.

Recall drills

What does RAG stand for in practice here?
Retrieve chunks → augment prompt with them → generate answer with LLM.
Lab 30 vs 31?
30: retriever returns documents only. 31: RetrievalQA returns an answer string.
ParentDocumentRetriever idea in one line?
Index small child chunks for search; return larger parent chunk for richer LLM context.
