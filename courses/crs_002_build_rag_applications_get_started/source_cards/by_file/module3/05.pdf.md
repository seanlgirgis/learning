# Part 1 — `config.py` (pages 9–12)

## Prompt templates (core of augmentation)

Placeholders:
- **`{context_str}`** — retrieved LinkedIn chunks
- **`{query_str}`** — user question

### `INITIAL_FACTS_TEMPLATE`
- **Purpose:** 3 interesting career/education facts
- **Rule:** Answer **only** from context
- Used by `generate_initial_facts()`

### `USER_QUESTION_TEMPLATE`
- **Purpose:** Answer specific profile questions
- **Rule:** Only from context; say **“I don’t know”** if missing

## Tunable RAG knobs

| Setting | Default idea | Effect |
|---------|--------------|--------|
| `CHUNK_SIZE` | 400–500 | Smaller = finer retrieval |
| `SIMILARITY_TOP_K` | e.g. 7 | More chunks = broader/noisier context |

**Test:** `python test_config.py` prints template flags + chunk settings.

# Part 2 — Concepts (pages 13–20)

## LLM / watsonx / LlamaIndex / RAG (page 13–14)

- **LLM:** Transformer, pretrained text generation
- **watsonx:** IBM AI suite; lab models: `ibm/granite-4-h-small`, `meta-llama/llama-3-3-70b-instruct`
- **LlamaIndex:** Data orchestration for genAI via RAG pipeline
- **RAG:** Your data not in LLM weights → load, index, retrieve, then generate

## Five RAG stages (pages 15–16) — lab uses first four

| Stage | LlamaIndex / lab |
|-------|------------------|
| **1 Loading** | `extract_linkedin_profile` → JSON dict |
| **2 Splitting** | `split_profile_data` → `Document` → `SentenceSplitter` → **nodes** |
| **3 Indexing** | Embed nodes → **vectors** (meaning numbers) |
| **4 Storing** | Inside `VectorStoreIndex` (in-memory in lab) |
| **5 Querying** | `query_engine.py` |
| *(6 Evaluation)* | Not implemented in lab |

**Index definition (page 15):** Data structure for fast query; built from **nodes**; becomes retriever + query engine.

**Embedding (page 15):** Watsonx embedding model; similar text → similar vectors.

## Icebreaker RAG flow (pages 16–17)

# Part 4 — `data_processing.py` (pages 27–32)

## Function 1: `split_profile_data(profile_data)`

```python
profile_json = json.dumps(profile_data)
document = Document(text=profile_json)
splitter = SentenceSplitter(chunk_size=config.CHUNK_SIZE)
nodes = splitter.get_nodes_from_documents([document])
```

**Why JSON string?** LinkedIn data is structured; whole profile as one text blob then split (~500 char chunks in prose).

**Maps to `01.md`:** Document → SentenceSplitter → nodes.

## Function 2: `create_vector_database(nodes)`

```python
embedding_model = create_watsonx_embedding()
index = VectorStoreIndex(
    nodes=nodes,
    embed_model=embedding_model,
    show_progress=True,
)
```

**Maps to `02.md`:** One step embed + store.

## Function 3: `verify_embeddings(index)`

- Walk node IDs in index
- Check vector store has non-None embedding per node
- Debug guard before querying

# Part 6 — `query_engine.py` (pages 39–45)

## What query engine does (page 39)

1. Retrieve from vector index  
2. Format prompt for LLM  
3. Generate coherent answer  

## `generate_initial_facts(index)`

```python
watsonx_llm = create_watsonx_llm(temperature=0.0, max_new_tokens=500, decoding_method="sample")
facts_prompt = PromptTemplate(template=config.INITIAL_FACTS_TEMPLATE)
query_engine = index.as_query_engine(
    streaming=False,
    similarity_top_k=config.SIMILARITY_TOP_K,
    llm=watsonx_llm,
    text_qa_template=facts_prompt,
)
response = query_engine.query("Provide three interesting facts...")
return response.response
```

## `answer_user_query(index, user_query)`

1. LLM: temperature=0, greedy, max 250 tokens  
2. `USER_QUESTION_TEMPLATE`  
3. **Manual retrieve** (teaching step): `base_retriever = index.as_retriever(...)` → `source_nodes`  
4. Build `context_str` from node text  
5. `as_query_engine` with `text_qa_template=question_prompt`  
6. `query_engine.query(user_query)` → return answer object  

**Note:** Shows both **retriever.retrieve** and **query_engine.query** — query engine still runs full pipeline internally.
