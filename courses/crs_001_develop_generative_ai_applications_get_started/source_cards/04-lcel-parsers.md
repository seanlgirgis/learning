- What does LCEL stand for? >>A)
    - LangChain Expression Language
    - Language Chain Execution Layer
    - Large Context Evaluation Logic
    - Linked Component Encoding Language
- What is the main purpose of LCEL? >>A)
    - Compose runnable components into reusable pipelines
    - Train a foundation model from scratch
    - Store vectors in a database
    - Create provider API keys
- What does `prompt | model | parser` represent? >>A)
    - A sequence that formats input, calls the model, and parses the output
    - Three unrelated variables
    - A database join
    - A model training loop
- In `prompt | model | parser`, what does the parser receive? >>A)
    - The model's output
    - Only the original input dictionary
    - The provider's API key
    - The prompt class definition
- Which call runs the complete LCEL pipeline? >>A)
    - `chain.invoke(input_data)`
    - `prompt.invoke(input_data)` only
    - `parser.to_string()`
    - `model.from_template()`
- Which call only formats the prompt when `prompt` is a prompt template? >>A)
    - `prompt.invoke(input_data)`
    - `chain.invoke(input_data)`
    - `model.invoke(input_data)`
    - `parser.invoke(input_data)`
- What is a RunnableSequence? >>A)
    - A workflow whose steps run one after another
    - A set of independent branches running from the same input
    - A vector database
    - A prompt variable
- Which workflow requires a sequence? >>A)
    - Translate text, then summarize the translation
    - Summarize a document and extract keywords independently
    - Classify the same document in two independent ways
    - Produce two unrelated outputs from the same input
- What is RunnableParallel? >>A)
    - A runnable that executes independent branches using the same input
    - A model that always returns several answers
    - A parser for parallel JSON arrays
    - A prompt that removes all variables
- Which workflow best fits RunnableParallel? >>A)
    - Summarize a document and independently extract keywords
    - Translate text and then summarize the translation
    - Retrieve documents and then use them in a prompt
    - Parse a model response and then validate it
- In `RunnableParallel(summary=summary_chain, keywords=keyword_chain)`, what do `summary` and `keywords` become? >>A)
    - Keys in the combined result dictionary
    - Input variables sent to the provider
    - API environment variable names
    - Class names generated by LangChain
- What is RunnableLambda used for? >>A)
    - Wrapping a regular Python function so it can participate in an LCEL pipeline
    - Selecting the cheapest model automatically
    - Storing prompts in a vector database
    - Converting every response into XML
- Why is automatic coercion useful in LCEL? >>A)
    - Ordinary functions or dictionaries can often be wrapped into runnable components
    - It changes the provider without testing
    - It removes the need for inputs
    - It guarantees identical model behavior
- What does StrOutputParser mainly do? >>A)
    - Extract the model's text into a plain Python string
    - Call the model
    - Build system and human messages
    - Count prompt tokens
- What is the final output of `prompt | model | StrOutputParser()` usually? >>A)
    - A plain string
    - A PromptTemplate object
    - A list of API keys
    - A vector embedding
- Why use a JSON output parser? >>A)
    - To transform model output into structured data that code can access by field
    - To reduce every prompt to one token
    - To replace the model provider
    - To prevent all possible errors
- Why is structured JSON often more useful to application code than plain text? >>A)
    - Code can reliably access named fields
    - It always costs less
    - It eliminates the need for validation
    - It works without a model
- Which is generally more reliable for structured data? >>A)
    - A defined schema or provider-supported structured-output mechanism
    - Asking for “some JSON” with no schema
    - Parsing arbitrary prose manually every time
    - Ignoring output format entirely
- What is the difference between an output parser and structured output? >>A)
    - A parser interprets a response after generation, while structured output constrains the response shape more directly
    - They are exactly the same in every provider
    - A parser changes model weights
    - Structured output only applies to images
- If StrOutputParser is replaced with JsonOutputParser, what can often remain unchanged? >>A)
    - The earlier prompt and model stages
    - The need for any model
    - The input variables
    - The provider credentials
- Why is LCEL composition valuable? >>A)
    - It reduces glue code and makes execution order clear
    - It forces all tasks to run in parallel
    - It removes the need to test providers
    - It guarantees zero token cost
- Which statement best describes provider neutrality in LangChain? >>A)
    - Much of the prompt-chain-parser structure can stay stable while the model adapter changes
    - Every provider supports every advanced feature identically
    - No retesting is needed when a provider changes
    - Provider credentials are unnecessary
- What should be retested when swapping providers? >>A)
    - Structured output, tools, streaming, metadata, and provider-specific behavior
    - Only the filename
    - Only the Python variable names
    - Nothing at all
- What does the whole chain become after LCEL composition? >>A)
    - A reusable runnable that can be invoked with new inputs
    - A permanent model training job
    - A database table
    - A static Markdown document
